# HMB Flutter Development

Flutter/Dart development skill for the Hold My Beer trade CRM app. Use when writing features, fixing bugs, modifying entities/DAOs, updating UI screens, or working with the FSM job lifecycle.

## Tech Stack

- **Framework**: Flutter (Dart SDK >=3.10.0 <4.0.0)
- **Database**: SQLite via sqflite (local-first, offline)
- **State**: Migrating from June/Provider mix → **Riverpod** (Phase 2)
- **Navigation**: `go_router`
- **FSM**: `fsm2` for job status lifecycle
- **Design**: Apple Cupertino design system (not Material)
- **Accounting**: Xero OAuth2 integration
- **AI**: Multi-provider via `AIProvider` abstraction (OpenRouter default, Claude, Ollama)
- **PDF**: `pdf` + `printing` for invoices/quotes
- **Error tracking**: Sentry
- **Linting**: `lint_hard` (strict)

## Architecture (Target)

```
lib/
  design_system/     # Apple design tokens, atoms, molecules, organisms
    tokens/          # colors.dart, typography.dart, spacing.dart, radius.dart
    atoms/           # app_icon, app_text, status_badge, avatar
    molecules/       # grouped_list_row, form_field_row, job_card, stat_card
    organisms/       # grouped_list, search_bar, action_sheet
    theme.dart       # ThemeExtension assembly
  entity/            # Data models (Customer, Job, Task, Invoice, Quote, etc.)
  dao/               # Data Access Objects — SQLite CRUD per entity
  service/           # Business logic layer (JobService, InvoiceService, QuoteService)
  ai/                # AIProvider abstraction + implementations
  database/          # DB factory, migration engine, backup providers
  fsm/               # Job status finite state machine
  api/               # External integrations (Xero, OAuth, booking sync)
  sync/              # Sync engine (outbox, push/pull, conflict resolution)
  ui/                # Screens organized by feature
  cache/             # Image caching and compression
  util/              # Utilities, parsers, Flutter helpers
```

## Key Architecture Decisions

1. **CupertinoApp root** — not MaterialApp. Apple design language everywhere
2. **Riverpod** for state management (replacing June + Provider + raw setState)
3. **Service layer** between UI and DAOs — business logic extracted from god-class DAOs
4. **Grouped inset lists** as primary layout pattern (iOS Settings-style)
5. **Inline validation** — errors below fields in systemRed, not toast-only
6. **Form dirty-state tracking** — "Unsaved changes" confirmation on back
7. **Search debouncing** — 300ms on all search fields
8. **Pagination** — virtual scrolling on all entity lists

## Database Migrations

- Raw SQL scripts in `assets/sql/upgrade_scripts/` (v1.sql through v164.sql)
- Manifest: `assets/sql/upgrade_list.json` (auto-generated by build tool)
- Migration engine runs scripts sequentially on app startup
- After adding a new migration: `dart run tool/build.dart --assets`

## Apple Design Principles

- 8pt spacing grid, 44pt minimum touch targets
- Semantic colors (label, secondaryLabel, separator, systemBackground)
- SF Pro typography scale (Large Title 34pt → Caption 11pt)
- Grouped list sections with ALL CAPS footnote headers
- Status badges: green=paid, yellow=sent, red=overdue, gray=draft
- Spring animations (250-350ms, critically damped)
- Light + dark mode from day one
- Progressive disclosure: summary on list, details on tap

## Coding Conventions

- 2-space indentation, 80-char line limit
- PascalCase for classes/enums, lowerCamelCase for members/locals
- Constants with leading `k`
- `dart format .` and `flutter analyze` must pass before commit
- Short imperative commit subjects
